## 一、当日P0任务（最终决策）

### P0-1: 基础技术架构搭建
**目标**: 建立可扩展的技术地基
**用户价值**: 支撑所有后续功能
**增长杠杆**: 开发效率 → 迭代速度
**验收标准**:
- [ ] 项目初始化 (Next.js + TypeScript + Tailwind)
- [ ] 数据库设计 (PostgreSQL schema)
- [ ] API框架搭建 (RESTful API结构)
- [ ] 基础CI/CD配置

### P0-2: 用户认证系统
**目标**: 用户注册/登录/微信授权
**用户价值**: 用户能创建账号，建立身份
**增长杠杆**: 留存 → 用户资产沉淀
**验收标准**:
- [ ] 手机号注册/登录
- [ ] 微信OAuth授权
- [ ] 用户角色区分（业主/租户）

### P0-3: 车位CRUD功能
**目标**: 业主能发布、查看、编辑、删除车位
**用户价值**: 供给侧核心，让闲置车位产生价值
**增长杠杆**: 供给 → 车位数量增长
**验收标准**:
- [ ] 车位信息录入（位置、价格、照片、可租时段）
- [ ] 车位列表展示
- [ ] 车位状态管理（可租/已租/下架）

### P0-4: 车位搜索+地图
**目标**: 租户能搜索和浏览附近车位
**用户价值**: 需求侧核心，让停车不再困难
**增长杠杆**: 转化 → 搜索到预订的转化
**验收标准**:
- [ ] 地图找车位（高德地图集成）
- [ ] 列表筛选（价格、距离、时段）
- [ ] 车位详情页

### P0-5: 订单预订系统
**目标**: 租户能预订车位，业主能确认/拒绝
**用户价值**: 交易闭环，完成核心价值验证
**增长杠杆**: GMV → 北极星指标
**验收标准**:
- [ ] 选择时段预订
- [ ] 业主确认/拒绝
- [ ] 订单状态管理
- [ ] 预订成功通知

## 四、关键假设与风险

### 假设
1. 能在1个试点社区获取20+种子车位
2. 业主愿意前期接受线下转账
3. 租户愿意前期无担保交易

### 风险
1. 🟠 监管风险：部分小区禁止车位转租
   - 缓解：选择政策友好社区试点
2. 🟠 冷启动：双边市场启动困难
   - 缓解：补贴种子业主，保底收益
3. 🟡 技术风险：地图API集成复杂度
   - 缓解：先用静态地图，逐步迭代
